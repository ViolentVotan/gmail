# ViewModels

State management layer between Services and Views. Each ViewModel uses the `@Observable` macro with default `MainActor` isolation.

## Guidelines

- ViewModels own mutable state as plain `var` properties (tracked automatically by `@Observable`). Views use `@State` to own ViewModel instances and `@Bindable` when they need write access to bindings.
- ViewModels do **not** import SwiftUI views. They may import `SwiftUI` for animations.
- One ViewModel per major screen/domain.
- **DB-first pattern**: `FullSyncEngine` syncs all messages to GRDB → `ValueObservation` drives the UI → folder switching is instant (zero API calls).
  - Database: per-account GRDB SQLite (`~/Library/Application Support/com.vikingz.vik.app/mail-db/`)
- **Optimistic mutations**: write to DB first → ValueObservation updates UI → API call → revert on failure via `restoreLabelsInDatabase`.
- ViewModels are **account-aware**: `accountID` is always a parameter or stored property.
- Conversion from API models to UI models happens here (e.g. `GmailMessage` -> `Email`).


## Files

| File | Role |
|------|------|
| `AppCoordinator.swift` | Navigation state (folder, selection, compose mode, pending draft selection, `selectionDirection: Edge` for directional detail transitions). Owns `MailDatabase` + `BackgroundSyncer` + `FullSyncEngine` lifecycle. Creates/starts engine on appear, stops/recreates on account switch (engine reference captured in local variable before nilling to ensure `stop()` runs). `accountSwitchGeneration` counter + `accountSwitchTask` guard prevent cascading `onChange` from cancelling account setup and handle rapid A→B→C switching safely (generation check at each suspension point, task-based guard in `handleFolderChange`). `lastRefreshedAt` tracks last sync completion, displayed in sidebar footer. Wires `SubscriptionsStore.analyze`, `updatePollingInterval` lifecycle. `handleSelectedEmailChange` cancels previous task. `selectPrevious()`/`selectNextEmail()` navigate to adjacent emails in `displayedEmails` (set `selectionDirection` before updating `selectedEmail` for directional transitions). `handleAccountChange` wraps UI state clearing in `withAnimation(VikAnimation.folderSwitch)` for smooth crossfade. Uses `Logger` (not print). `handleAccountChange` captures `oldID` before mutation, calls `loadContacts` (uses `[weak self]` in async `Task`), delegates to private `setupAccount` helper, cancels `navigationTask`. `handleAccountsChange(old:new:)` diffs old/new account arrays to detect adds/removes — loads per-account stores (`SnoozeStore`, `ScheduledSendStore`, `OfflineActionQueue`) for newly added accounts, stops engine and clears `SummaryService.accountID` on sign-out. Database deletion (`MailDatabase`, `AttachmentDatabase`) for all removed accounts runs in a stored `cleanupTask` that awaits the engine stop (if any) before deleting files, ensuring no writes to a deleted database. `SnoozeMonitor.shared.clearAllFailureCounts()` called on account removal to prevent orphaned failure-count entries. `SpotlightIndexer.shared.deleteAllItems()` called only when ALL accounts are removed (scoped to prevent destroying entries for still-signed-in accounts). `refreshSnoozedCacheIfNeeded()` / `refreshScheduledCacheIfNeeded()` conditionally refresh cached snooze/scheduled email arrays only when the respective folder is selected (guard on `selectedFolder`), driven by `SnoozeStore`/`ScheduledSendStore` count `onChange` in ContentView. Sets `mailboxViewModel.accountID` eagerly. `loadContacts()` called in `handleAppear()` (ensures contacts are available on first launch for single-account users). `loadCurrentFolder()` calls `syncEngine?.syncFolderIfEmpty(labelId:)` for lazy-loading Spam/Trash folders excluded from initial sync. `displayedEmails` is a stored `private(set) var` updated via `updateDisplayedEmails()` at all trigger points; `MailboxViewModel.onEmailsChanged` callback (wired in init, `[weak self]`) handles reactive DB/search updates. `navigateToMessage` stores a cancellable task with account guard after `await`. `navigateAndReply(gmailMessageID:replyAll:)` / `navigateAndForward(gmailMessageID:recipient:)` navigate then open compose with the appropriate mode (used by intent notification handlers). `pendingFolderChange` queues folder changes during account switch (applied in `handleAccountChange` defer block instead of silently dropping). `handleQuickReply` and `downloadAttachment` route service calls from ContentView. **Calendar integration**: `viewMode: AppViewMode` (`.mail` / `.calendar`) drives the top-level sidebar switcher and `⌘1`/`⌘2` menu commands. `calendarViewModel: CalendarViewModel` is created and owned here. `calendarSyncEngine` is started/stopped alongside the mail sync engine. `selectedCalendarEvent` holds the currently selected `CalendarEvent` for detail-panel routing. `miniAgendaEvents` is a cached array of today's upcoming events powering `MiniAgendaWidget`. `switchToCalendar()` / `switchToMail()` update `viewMode`. `navigateToEvent(_:)` sets `selectedCalendarEvent` and switches to calendar mode. `calendarNewEventTrigger` is a passthrough subject consumed by `CalendarContainerView` to open `CalendarEventEditorView`. `startCalendarSync()` / `stopCalendarSync()` manage `calendarSyncEngine` lifecycle in tandem with account changes. |
| `AuthViewModel.swift` | OAuth flow, account switching, sign-in/out state. Sign-out revokes OAuth token (best-effort), then explicitly cancels per-account scheduled work (`ScheduledSendStore`, `SnoozeStore`, `OfflineActionQueue`) and deletes token data (`TokenStore`) before calling `AccountStore.remove` (early calls are defense-in-depth to prevent a race with `SnoozeMonitor`). `MailDatabase` and `AttachmentDatabase` deletion is NOT done here — `AppCoordinator.handleAccountsChange` stops the sync engine first, then deletes the DB to avoid writes to a deleted database. Resets `isSignedIn` to `false` when the last account is removed, returning the user to `OnboardingView`. |
| `AttachmentStore.swift` | Attachment vault state, exclusion rules, progress tracking. |
| `CommandPaletteViewModel.swift` | Fuzzy-matched command search, keyboard navigation state. `filteredCommands` cached as stored property (updated via `updateFilteredCommands()`). In calendar mode, registers 7 calendar-specific commands (new event, go to today, day/week/agenda view switch, quick-add, edit event). When query text doesn't match any command, a dynamic "Create event: \<query\>" quick-add entry is appended, dispatching to `CalendarViewModel.quickAddEvent`. |
| `ComposeViewModel.swift` | Draft management, send, auto-save, inline images, Bcc. `parentMessageID` and `parentReferences` store threading context for reply chain construction. `wasScheduled` flag tracks schedule-send state. Reply orchestration (`sendReplyMessage`, `loadExistingDraft`, `scheduleReplyAutoSave`) — `sendReplyMessage` accepts `parentMessageID`/`parentReferences` for proper References header threading; `scheduleReply` flow uses these for scheduled reply delivery. `send()` captures all parameters by value in the undo `onConfirm` closure (survives view dismissal — calls `GmailSendService.shared.send()` directly). `scheduleSend(at:)` persists reply draft cleanup (`saveReplyDrafts()`) and cleans `gmailDrafts`, matching `send()`. `scheduleReplyAutoSave` uses `[weak self]` in its debounced `Task`. `saveDraft` passes `bcc` to draft service (create and update). `openAttachmentPicker() async -> [URL]` uses NSOpenPanel async API. Shared file-drop logic. |
| `ComposeModeInitializer.swift` | Initializes compose fields based on mode (reply, forward, new). `ComposeModeFields` struct includes `parentMessageID` and `parentReferences` for threading header propagation. `apply(mode:...)` extracts these from `ComposeMode.reply`/`.replyAll` cases. |
| `EmailActionCoordinator.swift` | Email mutations (archive, delete, star, read, spam, labels, unsnooze, print) with offline queue support. `markReadEmail` mirrors `markUnreadEmail` with offline support. All public action methods are `async` (callers wrap in `Task { await ... }`). Shared mutation path via private `performUndoableAction` (single) and `performBulkUndoableAction` (bulk — generalized optimistic-update + undo pattern). Bulk methods use `updateLabelsInDatabaseBatch` for single-transaction optimistic updates (one `ValueObservation` notification instead of N). `bulkMarkUnread`/`bulkMarkRead` revert optimistic updates on API failure via `restoreLabelsInDatabase`. `bulkMoveToInbox` guards on `NetworkMonitor.shared.isConnected` (offline shows error toast). `printEmail` routes through the coordinator (not direct service calls from views). All undo closures use `[weak vm]` captures and guard on `accountID` to prevent retain cycles during account switch. `confirmEmptyFolder` uses `[weak self]` in its async `Task`. |
| `EmailDetailViewModel.swift` | Thread loading with DB fast path (fires `onMessagesRead` for unread badge sync) then API refresh (skips duplicate tracker analysis when data unchanged), attachment download. `backgroundTasks` is `Mutex<[Task]>` for safe `deinit` cancellation. Business logic: compose mode construction (reply/replyAll/forward), attachment transforms, label suggestion application. Tracker sanitization via `@concurrent sanitizeOffMainActor`. Smart reply suggestions (`smartReplySuggestions` property, `loadSmartReplies` spawns a `Task { @MainActor [weak self] }` to deliver results on the main actor). **Calendar context**: `matchedCalendarEvent` and `calendarContextEvent: CalendarEvent?` are set by `detectCalendarContext()`, which queries `CalendarIntegrationService` to find an event whose attendees include the email's sender — driving the `CalendarContextCard` in `EmailDetailView`. |
| `EmailSummaryViewModel.swift` | Apple Foundation Models email summary generation with streaming support. `startStreaming` cancels any in-flight `streamTask`/`insightTask` before starting new ones. |
| `FiltersViewModel.swift` | Gmail filters state: load, create, delete. `deleteFilter` shows toast on failure. Integrates with `GmailFilterService`. Account-aware. |
| `MailStore.swift` | `@Observable @MainActor` local draft store, Gmail draft sync, reply draft persistence (`ReplyDraftInfo`). `syncGmailDrafts` fetches drafts from Gmail API and backfills `gmail_draft_id` into the GRDB database (required by `SendDraftIntent`). Drafts folder result (`emails(for: .drafts)`) cached via `_cachedDrafts`, invalidated by `didSet` on `emails` and `gmailDrafts`. Uses `Logger` (not print). |
| `MailboxViewModel.swift` | `userLabels` cached property (filtered non-system labels, updated via `labels` `didSet`). Email list driven entirely by GRDB `ValueObservation` (`.tracking` — not `trackingConstantRegion` because the JOIN through labels varies per folder — + async `for try await values` API for guaranteed MainActor delivery via `observationTask`) — folder switching starts a DB observation, no API calls. `onEmailsChanged` callback (`@ObservationIgnored`) fires on `emails` `didSet` for reactive updates. DB-first optimistic mutations via shared `performOptimisticAction` helper (handles optimistic DB write → API call → revert + toast on failure; used by markAsUnread, toggleStar, trash, archive, moveToInbox, untrash, unspam, spam). `markAsRead` and `deletePermanently` remain manual (combined flag update and record deletion respectively). `addLabel`/`removeLabel` support offline via `OfflineActionQueue`. `applyReadLocally` is `async` and batches all updates in a single `dbPool.write` transaction. `updateLabelsInDatabaseBatch` performs bulk label mutations in a single transaction (returns original labels map for undo). `updateDockBadge` is `static async` — uses async `dbPool.read` to avoid blocking the main thread. All 5 label mutation methods refactored into a shared `writeLabels(_:ensureLabelRecords:transform:)` `async` helper that syncs `is_read`/`is_starred` denormalized columns. `loadCategoryUnreadCounts` uses parameterized SQL (no string interpolation). `enrichmentTask` uses `[weak self]` + DB identity guard, cancelled on account switch. `threadedEmails` uses `max(by:)` instead of `sorted.first`. FTS5 local search. `deleteLabel` removes the label from the in-memory array, calls the Gmail API, then deletes the `LabelRecord` from the database (triggering `ON DELETE CASCADE` to clean `message_labels` join rows). `deletePermanently(_:originalLabelIds:)` deletes the message record (via `BackgroundSyncer` or direct write); `originalLabelIds` is passed through to the undo closure for label restoration. Uses `Logger` (not print). |
| `PanelCoordinator.swift` | Side panel state (shortcuts, debug, original message, attachments, browser). Uses `VikAnimation.springDefault`/`.springSnappy` instead of inline spring values. `showOriginalMessage` and `downloadMessage` accept injectable `fetchRaw` closure (defaults to `GmailMessageService.shared.getRawMessage`) for testability. `downloadTask` stored and cancelled on re-entry to prevent concurrent downloads. |
| `SyncProgressManager.swift` | Sync UI progress state (bubble visibility, debounce/linger timers). `SyncPhase` enum: idle, initialSync(synced, estimated), bodyPrefetch(remaining), syncing, success, error. Account-aware, environment-injected. |
| `CalendarViewModel.swift` | `@Observable @MainActor`. Drives all calendar UI. State: `viewMode: CalendarViewMode` (week/day/agenda), `selectedDate: Date`, `events: [CalendarEvent]`, `calendars: [CalendarInfo]`, `visibleCalendarIDs: Set<String>`. `ValueObservation` on the `calendar_events` and `calendars` GRDB tables keeps the UI reactive to background sync writes. Navigation: `navigateForward()`, `navigateBack()`, `goToToday()`. CRUD: `createEvent(_:)`, `updateEvent(_:scope:)`, `deleteEvent(_:scope:)`, `respondToEvent(_:status:)`, `quickAddEvent(text:)` — all methods fall back to `CalendarOfflineActionQueue` when offline. Computed helpers: `selectedWeek: [Date]`, `eventsForDay(_:) -> [CalendarEvent]`, `eventsForTimeSlot(day:hour:) -> [CalendarEvent]`, `hasConflict(event:) -> Bool`. |
| `EventDetailViewModel.swift` | `@Observable @MainActor`. Manages event detail and inline editing. Holds an `EventEditDraft` for in-progress edits. `saveChanges()` detects etag conflicts (sets `ConflictState.detected`) and exposes `resolveConflict(strategy:)`. `joinConference()` opens the conference URL via `NSWorkspace.open`. `findRelatedEmails()` is stubbed, pending `CalendarIntegrationService` wiring to surface emails from/to the event's attendees. |
