# ViewModels

State management layer between Services and Views. Each ViewModel uses the `@Observable` macro with default `MainActor` isolation.

## Guidelines

- ViewModels own mutable state as plain `var` properties (tracked automatically by `@Observable`). Views use `@State` to own ViewModel instances and `@Bindable` when they need write access to bindings.
- ViewModels do **not** import SwiftUI views. They may import `SwiftUI` for animations.
- One ViewModel per major screen/domain.
- **DB-first pattern**: `FullSyncEngine` syncs all messages to GRDB → `ValueObservation` drives the UI → folder switching is instant (zero API calls).
  - Database: per-account GRDB SQLite (`~/Library/Application Support/com.vikingz.vik.app/mail-db/`)
- **Optimistic mutations**: write to DB first → ValueObservation updates UI → API call → revert on failure via `restoreLabelsInDatabase`.
- ViewModels are **account-aware**: `accountID` is always a parameter or stored property.
- Conversion from API models to UI models happens here (e.g. `GmailMessage` -> `Email`).


## Files

| File | Role |
|------|------|
| `AppCoordinator.swift` | Thin orchestrator owning 6 domain sub-coordinators (`navigation`, `selection`, `compose`, `calendar`, `dialogs`, `sync`) + 7 child VMs. Cross-domain lifecycle handlers (`handleAppear`, `setupAccount`, `handleFolderChange`, `handleLabelChange`, `handleCategoryChange`, `handleAccountChange`, `handleAccountsChange`) coordinate across sub-coordinators. Cross-domain compose methods (`composeNewEmail`, `startCompose`, `discardDraft`) span navigation + selection + compose. Navigation actions (`navigateToMessage`, `navigateAndReply`, `navigateAndForward`) fetch messages and route to panel/compose. Service routing (`handleQuickReply` delegates to `@concurrent sendQuickReply`, `downloadAttachment`, `previewToggleStar`, `previewMarkUnread`). Cross-domain computed properties: `listIsLoading` (reads navigation + mailStore + mailboxVM), `isComposeActive` (reads navigation + selection). Convenience wrappers pass cross-coordinator params: `switchToCalendar()` passes `sync.mailDatabase` to `CalendarCoordinator`, `loadContacts()` passes `navigation.accountID` to `SyncCoordinator`. `updateDisplayedEmails()` routes `navigation.selectedFolder` + cached emails from `sync` to `selection.updateDisplayedEmails()`. Wires `SyncCoordinator.onCacheRefreshed` and `MailboxViewModel.onEmailsChanged` callbacks to trigger `updateDisplayedEmails()`. `handleAccountChange` wraps UI state clearing in `withAnimation(VikAnimation.folderSwitch)`. `handleAccountsChange(old:new:)` diffs old/new account arrays for add/remove handling. Uses `Logger` (not print). |
| `NavigationCoordinator.swift` | Folder, category, label, and account selection state. Properties: `selectedAccountID`, `selectedFolder`, `selectedInboxCategory`, `selectedLabel`, `searchResetTrigger`, `searchFocusTrigger`. Computed: `accountID` (falls back to `authViewModel.primaryAccount`), `fromAddress` (looks up email in `authViewModel.accounts`). Holds `AuthViewModel` reference for computed properties. |
| `SelectionCoordinator.swift` | Email selection state and displayed emails cache. Properties: `selectedEmail`, `selectedEmailIDs`, `selectionDirection`, `displayedEmails` (private(set)). Methods: `selectNext`, `selectPrevious`, `selectNextEmail`, `clearSelection`, `deselectAll`, `selectAllEmails`. `updateDisplayedEmails(folder:mailStore:mailboxViewModel:cachedSnoozedEmails:cachedScheduledEmails:)` takes all data sources as parameters (no circular deps). `handleSelectedEmailChange` marks as read via `MailboxViewModel` and indexes in Spotlight. Holds `MailboxViewModel` reference. |
| `ComposeCoordinator.swift` | Compose mode, signatures, and draft selection staging. Properties: `composeMode`, `signatureForNew`, `signatureForReply`. Private: `pendingDraftSelection` with `setPendingDraftSelection`/`consumePendingDraftSelection` accessors. Methods: `loadSignatures(for:)`, `saveSignatures(for:)` (UserDefaults persistence). |
| `CalendarCoordinator.swift` | Calendar view mode, CalendarViewModel lifecycle, sync engine, mini-agenda. Properties: `viewMode` (mail/calendar), `calendarViewModel` (private(set)), `calendarSyncEngine` (private(set)), `miniAgendaEvents`, `calendarNewEventTrigger`. Methods: `switchToCalendar(db:)` (creates CalendarViewModel, wires `onEventMutated`), `switchToMail()`, `loadMiniAgendaEvents(db:accountID:)`, `navigateToEvent(_:db:)`, `startCalendarSync(for:db:)`, `stopCalendarSync()`. `clearState()` nils VM + sync engine for account switch. Methods take `db` parameter from SyncCoordinator (no circular dependency). |
| `DialogCoordinator.swift` | Confirmation dialog state. Properties: `showEmptyTrashConfirm`, `trashTotalCount`, `showEmptySpamConfirm`, `spamTotalCount`. Methods: `emptyTrashRequested(count:)`, `emptySpamRequested(count:)`. Zero dependencies. |
| `SyncCoordinator.swift` | Database lifecycle, sync engines, background syncer, contacts, snoozed/scheduled caches. Properties: `mailDatabase` (private(set)), `backgroundSyncer` (private(set)), `syncEngine` (private(set)), `attachmentIndexer`, `contacts` (private(set)), `undoDuration`, `snoozedEmails`/`scheduledEmails` (cached). Private task management: `lifecycleTask`, `contactsTask`, `cleanupTask`, `accountSwitchTask`, `accountSwitchGeneration`. Methods: `openDatabase(for:)` (`@concurrent`), `setupDatabase(for:selectedAccountID:syncProgressManager:)`, `loadContacts(accountID:)`, `refreshSnoozedCache(fromAddress:)`/`refreshScheduledCache(fromAddress:)` (call `onCacheRefreshed` callback after). `onCacheRefreshed: (() -> Void)?` callback for cross-coordinator notification. Setter/clear methods for engine lifecycle. `cancelAllTasks()` for deinit cleanup. |
| `AuthViewModel.swift` | OAuth flow, account switching, sign-in/out state. Sign-out revokes OAuth token (best-effort), then explicitly cancels per-account scheduled work (`ScheduledSendStore`, `SnoozeStore`, `OfflineActionQueue`) and deletes token data (`TokenStore`) before calling `AccountStore.remove` (early calls are defense-in-depth to prevent a race with `SnoozeMonitor`). `MailDatabase` and `AttachmentDatabase` deletion is NOT done here — `AppCoordinator.handleAccountsChange` stops the sync engine first, then deletes the DB to avoid writes to a deleted database. Resets `isSignedIn` to `false` when the last account is removed, returning the user to `OnboardingView`. |
| `AttachmentStore.swift` | Attachment vault state, exclusion rules, progress tracking. |
| `CommandPaletteViewModel.swift` | Fuzzy-matched command search, keyboard navigation state. `filteredCommands` cached as stored property (updated via `updateFilteredCommands()`). In calendar mode, registers 7 calendar-specific commands (new event, go to today, day/week/agenda view switch, quick-add, edit event). When query text doesn't match any command, a dynamic "Create event: \<query\>" quick-add entry is appended, dispatching to `CalendarViewModel.quickAddEvent`. |
| `ComposeViewModel.swift` | **Single source of truth** for all compose state (standalone compose + inline reply bar). Draft management, send, auto-save, inline images, Bcc. Threading: `parentMessageID`/`parentReferences` for reply chain construction. `send()` captures all parameters by value in the undo `onConfirm` closure (survives view dismissal). **Reply bar capabilities** (formerly in deleted `ReplyBarManager`): `showCc`/`showBcc`, `subjectOverride`, `sendError`, `quickReplies`, `isLoadingDraft`/`isLoadingReplies`, `collapsedPlaceholderText` (cached with recipient context — "Reply to {name}" / "Reply All ({n})" / "Draft: ..."). Lifecycle: `resetForEmail(_:onGenerateQuickReplies:)`, `collapse(email:mailStore:)`. Send: `sendReplyFromBar(email:editorInlineImages:mailStore:)`, `scheduleReplyFromBar(at:email:editorInlineImages:mailStore:)`. Auto-save: `scheduleReplyAutoSaveUnified(email:mailStore:)` with local-first persistence (writes `mailStore.replyDrafts` immediately, rate-limits API save at 10s intervals). Draft load: `loadExistingDraftForReply(email:mailStore:loader:editorState:)` with generation guards. Content: `updateCachedText(html:)` with 150ms debounce. File handling: `handleFileDropForReply(_:editorState:)`, `attachFilesForReply()`. |
| `ComposeModeInitializer.swift` | Initializes compose fields based on mode (reply, forward, new, newTo). `ComposeModeFields` struct includes `parentMessageID` and `parentReferences` for threading header propagation. `apply(mode:...)` extracts these from `ComposeMode.reply`/`.replyAll` cases; `.newTo` pre-fills `fields.to` with the target email. Shared `applyQuotedBody` and `applyThreading` helpers eliminate duplication across reply/replyAll/forward paths. |
| `ContactPopoverViewModel.swift` | `@Observable @MainActor` VM for the contact info popover. Queries GRDB `ContactRecord` by email for known-contact detection, enriches via `PeopleAPIService.fetchPersonDetails` with a 5-minute in-memory cache (keyed by `accountID:email`). For unknown senders, exposes header-derived properties (`sentByDomain`, `mailedBy`, `signedBy`, `encryptionInfo`, `isSuspiciousSender`) from `GmailMessage`. Actions: `copyEmail`, `composeEmail`, `searchEmails`, `openContact`, `addToContacts`. |
| `EmailActionCoordinator.swift` | Email mutations (archive, delete, star, read, spam, labels, unsnooze, print) with offline queue support. `markReadEmail` mirrors `markUnreadEmail` with offline support. All public action methods are `async` (callers wrap in `Task { await ... }`). Shared mutation path via private `performUndoableAction` (single) and `performBulkUndoableAction` (bulk — generalized optimistic-update + undo pattern). Bulk methods use `updateLabelsInDatabaseBatch` for single-transaction optimistic updates (one `ValueObservation` notification instead of N). `bulkMarkUnread`/`bulkMarkRead` revert optimistic updates on API failure via `restoreLabelsInDatabase`. `bulkMoveToInbox` guards on `NetworkMonitor.shared.isConnected` (offline shows error toast). `printEmail` routes through the coordinator (not direct service calls from views). All undo closures use `[weak vm]` captures and guard on `accountID` to prevent retain cycles during account switch. `confirmEmptyFolder` uses `[weak self]` in its async `Task`. |
| `EmailDetailViewModel.swift` | Thread loading with DB fast path (fires `onMessagesRead` for unread badge sync) then API refresh (skips duplicate tracker analysis when data unchanged), attachment download. `backgroundTasks` is `Mutex<[Task]>` for safe `deinit` cancellation. Business logic: compose mode construction (reply/replyAll/forward), attachment transforms, label suggestion application. Tracker sanitization via `@concurrent sanitizeOffMainActor`. Smart reply suggestions (`smartReplySuggestions` property, `loadSmartReplies` spawns a `Task { @MainActor [weak self] }` to deliver results on the main actor). **Calendar context**: `matchedCalendarEvent` and `calendarContextEvent: CalendarEvent?` are set by `detectCalendarContext()`, which queries `CalendarIntegrationService` to find an event whose attendees include the email's sender — driving the `CalendarContextCard` in `EmailDetailView`. |
| `EmailSummaryViewModel.swift` | Apple Foundation Models email summary generation with streaming support. `startStreaming` cancels any in-flight `streamTask`/`insightTask` before starting new ones. |
| `FiltersViewModel.swift` | Gmail filters state: load, create, delete. `deleteFilter` shows toast on failure. Integrates with `GmailFilterService`. Account-aware. |
| `MailStore.swift` | `@Observable @MainActor` local draft store, Gmail draft sync, reply draft persistence (`ReplyDraftInfo` with `gmailDraftID`, `preview`, `lastModified` — custom `init(from:)` decodes legacy JSON without `lastModified` as `.distantPast`). `syncGmailDrafts` fetches drafts from Gmail API and backfills `gmail_draft_id` into the GRDB database (required by `SendDraftIntent`). Drafts folder result (`emails(for: .drafts)`) cached via `_cachedDrafts`, invalidated by `didSet` on `emails` and `gmailDrafts`. Uses `Logger` (not print). |
| ~~`ReplyBarManager.swift`~~ | **Deleted.** Reply-bar state and logic unified into `ComposeViewModel` — see its `// MARK: - Reply Bar` sections. `ReplyBarView` now binds directly to `ComposeViewModel` via `@State`. |
| `MailboxViewModel.swift` | `userLabels` cached property (filtered non-system labels, updated via `labels` `didSet`). Email list driven entirely by GRDB `ValueObservation` (`.tracking` — not `trackingConstantRegion` because the JOIN through labels varies per folder — + async `for try await values` API for guaranteed MainActor delivery via `observationTask`) — folder switching starts a DB observation, no API calls. `onEmailsChanged` callback (`@ObservationIgnored`) fires on `emails` `didSet` for reactive updates. DB-first optimistic mutations via shared `performOptimisticAction` helper (handles optimistic DB write → API call → revert + toast on failure; used by markAsUnread, toggleStar, trash, archive, moveToInbox, untrash, unspam, spam). `markAsRead` and `deletePermanently` remain manual (combined flag update and record deletion respectively). `addLabel`/`removeLabel` support offline via `OfflineActionQueue`. `applyReadLocally` is `async` and batches all updates in a single `dbPool.write` transaction. `updateLabelsInDatabaseBatch` performs bulk label mutations in a single transaction (returns original labels map for undo). `updateDockBadge` is `static async` — uses async `dbPool.read` to avoid blocking the main thread. All 5 label mutation methods refactored into a shared `writeLabels(_:ensureLabelRecords:transform:)` `async` helper that syncs `is_read`/`is_starred` denormalized columns. `loadCategoryUnreadCounts` uses parameterized SQL (no string interpolation). `enrichmentTask` uses `[weak self]` + DB identity guard, cancelled on account switch. `threadedEmails` uses `max(by:)` instead of `sorted.first`. FTS5 local search. `deleteLabel` removes the label from the in-memory array, calls the Gmail API, then deletes the `LabelRecord` from the database (triggering `ON DELETE CASCADE` to clean `message_labels` join rows). `deletePermanently(_:originalLabelIds:)` deletes the message record (via `BackgroundSyncer` or direct write); `originalLabelIds` is passed through to the undo closure for label restoration. Uses `Logger` (not print). |
| `PanelCoordinator.swift` | Side panel state (shortcuts, debug, original message, attachments, browser). Uses `VikAnimation.springDefault`/`.springSnappy` instead of inline spring values. `showOriginalMessage` and `downloadMessage` accept injectable `fetchRaw` closure (defaults to `GmailMessageService.shared.getRawMessage`) for testability. `downloadTask` stored and cancelled on re-entry to prevent concurrent downloads. |
| `SyncProgressManager.swift` | Sync UI progress state (bubble visibility, debounce/linger timers). `SyncPhase` enum: idle, initialSync(synced, estimated), bodyPrefetch(remaining), syncing, success, error. Account-aware, environment-injected. |
| `CalendarViewModel.swift` | `@Observable @MainActor`. Drives all calendar UI. State: `viewMode: CalendarViewMode` (default `.month`), `selectedDate: Date`, `events: [CalendarEvent]`, `calendars: [CalendarInfo]`, `visibleCalendarIDs: Set<String>`. `ValueObservation` on the `calendar_events` and `calendars` GRDB tables keeps the UI reactive to background sync writes. Navigation: `navigateForward()`, `navigateBack()`, `goToToday()` — month mode steps ±1 month, date range spans 42 days (6 complete weeks Mon–Sun). CRUD: `createEvent(_:)`, `updateEvent(_:scope:)`, `deleteEvent(_:scope:)`, `respondToEvent(_:status:)`, `quickAddEvent(text:)` — all methods fall back to `CalendarOfflineActionQueue` when offline. Cached helpers (stored properties recomputed via `recomputeEventCaches()` when `events` changes): `selectedWeek: DateInterval` (recomputed via `recomputeSelectedWeek()` on navigation), `eventsByDay: [Date: [CalendarEvent]]` (pre-grouped dictionary for O(1) day lookup — used by month view), `multiDayEvents: [CalendarEvent]` (events spanning more than one day — used for spanning bar computation). `eventsForDay(_:) -> [CalendarEvent]` uses O(1) dictionary lookup into cached `eventsByDay`. Other computed: `eventsForTimeSlot(day:hour:) -> [CalendarEvent]`, `hasConflict(event:) -> Bool`. `onEventMutated: (() async -> Void)?` callback (wired by `AppCoordinator`) fires after all successful mutations. |
