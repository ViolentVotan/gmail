# Serif — Swift 6.2 & macOS 26 Migration Design

**Date:** 2026-03-11
**Approach:** Big Bang (single feature branch, all changes at once)
**Target:** Swift 6.2, macOS 26+, Xcode 26
**Baseline:** Verified clean build on current codebase (Swift 5.0, macOS 14+, Xcode 26.3)

---

## 1. Project Configuration & Build Settings

### Changes
- `SWIFT_VERSION` → `6.2` (from `5.0`)
- `MACOSX_DEPLOYMENT_TARGET` → `26.0` (from `14.0`)
- Enable `SWIFT_APPROACHABLE_CONCURRENCY` build setting — makes `@MainActor` the default isolation for the entire module
- Swift 6 language mode enables strict concurrency checking by default
- Update SPM dependencies (AppAuth, Sparkle, BlossomColorPicker) to latest macOS 26-compatible versions

### Dead Code Removal
- Remove `APICache` class from `APICache.swift` (DEBUG-only disk cache — not used in production). **Keep `APILogger` and `APILogEntry`** in the same file — they are used unconditionally by `GmailAPIClient.rawRequest` and `DebugMenuView`
- Remove `#available` / `@available` checks for **macOS 14/15 only** — minimum is now 26. **Keep `#available(macOS 26.0, *)` guards** on FoundationModels (Apple Intelligence) calls in `QuickReplyService`, `SummaryService`, `LabelSuggestionService`, `ContentView`, `EmailRowView`, `EmailHoverSummaryView` — these guard runtime feature availability, not OS backcompat
- Remove any backward-compatibility shims or fallback code paths for older macOS versions

## 2. Observation Migration (ObservableObject → @Observable)

The largest single change. Touches all ObservableObject types and most of the 58 Views.

### All ObservableObject Types (~17 classes)

**ViewModels folder (9 files):** AppCoordinator, AuthViewModel, MailboxViewModel, ComposeViewModel, EmailDetailViewModel, EmailActionCoordinator, PanelCoordinator, UpdaterViewModel, AttachmentStore

Note: `ComposeModeInitializer` is a plain struct — no migration needed.

**Outside ViewModels folder (11 types):**
- `MailStore` (Models/) — draft persistence
- `ThemeManager` (Theme/) — current theme state, injected via environment into all views
- `WebRichTextEditorState` (Views/Common/) — JS↔SwiftUI bridge (**special handling**, see below)
- `SwipeRowState` (Views/EmailList/SwipeableEmailRow.swift) — swipe gesture state
- `SwipeCoordinator` (Views/EmailList/SwipeableEmailRow.swift) — swipe coordinator
- `ToastManager` (Services/) — toast notifications
- `NetworkMonitor` (Services/) — connectivity state
- `APILogger` (Services/) — API debug logging
- `ThumbnailCache` (Services/) — image cache
- `SubscriptionsStore` (Services/) — subscription feed state
- `UndoActionManager` (Services/) — undo toast state
- `WebViewStore` (private class in InAppBrowserView.swift) — browser state

**Total: ~20 ObservableObject types**

For each: remove `ObservableObject` conformance, add `@Observable` macro
- Remove all `@Published` property wrappers (~103 occurrences) — properties become plain `var`
- Remove all Combine `objectWillChange` forwarding sinks (nested VM change propagation pattern)
- Remove `import Combine` where it was only used for observation plumbing
- Remove `cancellables: Set<AnyCancellable>` storage bags used for forwarding
- Remove any manual `objectWillChange.send()` calls

### Views (58 files)
| Old Pattern | New Pattern | Count |
|---|---|---|
| `@StateObject var vm = VM()` | `@State var vm = VM()` | ~13 |
| `@ObservedObject var vm` | `var vm` or `@Bindable var vm` | many |
| `@EnvironmentObject var x: T` | `@Environment(T.self) var x` | few |
| `.environmentObject(x)` | `.environment(x)` | few |

### @Bindable
Where views use `$viewModel.property` for two-way bindings (~102 `@Binding` sites), the owning view needs `@Bindable var viewModel` to produce bindings from `@Observable` objects.

### Special Case: WebRichTextEditorState
`WebRichTextEditorState` is passed via `@ObservedObject` into `WebRichTextEditorRepresentable` (an `NSViewRepresentable`). `NSViewRepresentable` does not participate in `@Observable` observation — `updateNSView` won't trigger from `@Observable` property changes automatically. This requires either:
- Keeping `WebRichTextEditorState` as `ObservableObject` (exception to the migration), OR
- Adding manual observation via `withObservationTracking` in the representable's coordinator to detect changes and trigger updates
The recommended approach is to keep it as `ObservableObject` for now — it's a JS bridge, not a typical ViewModel.

### Dead Code Removal
- All Combine-based observation forwarding code
- `AnyCancellable` storage used purely for observation
- Any custom `ObservableObject` extensions or helpers
- `import Combine` statements where Combine is no longer used

## 3. Concurrency Modernization (Swift 6.2 Approachable Concurrency)

With `SWIFT_APPROACHABLE_CONCURRENCY` enabled, the entire module defaults to `@MainActor`.

### Remove Redundant @MainActor (~40 occurrences)
- All ViewModels marked `@MainActor final class` — annotation becomes redundant
- All View-layer functions marked `@MainActor` — redundant
- Closures marked `@MainActor` for UI work — redundant

### Add @concurrent for Background Work
With approachable concurrency, all functions default to `@MainActor`. Service methods that do I/O or heavy computation must explicitly opt out to avoid blocking the main thread. **Only annotate methods that would otherwise be implicitly `@MainActor` — methods already in `nonisolated` types don't need it.**

- Service methods doing network I/O (GmailAPIClient, GmailMessageService, etc.) → `@concurrent`
- `withThrowingTaskGroup` batch operations with API calls → `@concurrent` task closures
- Heavy computation (ContentExtractor HTML parsing, TrackingBlocker analysis) → `@concurrent`
- **Caution:** `@concurrent async` functions cannot access `@MainActor`-isolated state directly. Audit each callsite — if a service method reads/writes MainActor state, restructure to separate the I/O from the state mutation.

### Sendable Conformances (now enforced)
- All model structs (`Email`, `GmailAccount`, `ComposeMode`, `IndexedAttachment`, `EmailLabel`, `GmailMessage`, etc.) → explicit `Sendable`
- API response types → `Sendable`
- Use `sending` keyword for closures crossing isolation boundaries

### Replace DispatchQueue with Actor Isolation
- Combine `.receive(on: DispatchQueue.main)` — gone (removed with observation migration)
- Any remaining `DispatchQueue.main.async` → rely on default MainActor isolation
- Any `DispatchQueue.global()` → `@concurrent` functions or `Task.detached`

### Fix Unsafe Patterns
- `[unowned self]` in async closures → audit, replace with `[weak self]` or structured concurrency
- `@unchecked Sendable` (~18 occurrences) → audit each, make properly Sendable or use actors

### Dead Code Removal
- Explicit `@MainActor` annotations (redundant under approachable concurrency)
- `DispatchQueue.main` / `DispatchQueue.global()` usage
- `@unchecked Sendable` workarounds no longer needed

## 4. SwiftUI Modernization (macOS 26 Features)

### Native WebView (partial — HTMLEmailView only)
- Replace `HTMLEmailView`'s `WKWebView` + `NSViewRepresentable` with SwiftUI `WebView` + `WebPage` model
- Fixes known security issue (overly permissive WKNavigationDelegate policy) in email reading
- Email HTML rendering in `EmailDetailView` migrates to native `WebView`

**Components that STAY as WKWebView (cannot migrate):**
- `WebRichTextEditorRepresentable` — bidirectional JS-driven rich text editor using `evaluateJavaScript`, `WKScriptMessageHandler("editor")`, drag-and-drop, custom HTML templates. Native `WebView` has no JS evaluation or script message handler API.
- `InAppBrowserView` — uses `NSKeyValueObservation` on `WKWebView.isLoading`, `canGoBack`/`canGoForward` state, which `WebPage` API may not expose directly.

### Liquid Glass
- Toolbars, navigation bars, sidebars get Liquid Glass automatically when compiled with Xcode 26
- Evaluate 15 custom themes against Liquid Glass — adjust where glass effects conflict with theme colors
- Apply `.glassEffect()` to custom overlay views (ToastOverlayView, SlidePanelsOverlay) where appropriate

### New SwiftUI APIs
| API | Replaces | Where |
|-----|----------|-------|
| `ContentUnavailableView` | Custom empty state views | Empty mailbox, no search results, offline |
| `onChange(of:initial:)` | Old single-param `onChange` | All views with `onChange` |
| `#Preview` macro | `PreviewProvider` structs | All preview code |
| `@Entry` macro | Manual `EnvironmentKey` | Custom theme environment value |
| `.symbolEffect` | Custom loading indicators | Refresh, loading states |

### Dead Code Removal
- `HTMLEmailView`'s `WKWebView` wrapper, `NSViewRepresentable` bridge, `WKNavigationDelegate` code (keep WKWebView in `WebRichTextEditorRepresentable` and `InAppBrowserView`)
- Custom empty state views replaced by `ContentUnavailableView`
- Manual `EnvironmentKey` boilerplate replaced by `@Entry`
- Old `PreviewProvider` structs
- Old single-parameter `onChange` calls

## 5. Testing Migration (XCTest → Swift Testing)

Migrate all 11 test files (~2,726 lines).

### Mechanical Conversions
| Old (XCTest) | New (Swift Testing) |
|---|---|
| `import XCTest` | `import Testing` |
| `class FooTests: XCTestCase` | `struct FooTests` |
| `func testSomething()` | `@Test func something()` |
| `XCTAssertEqual(a, b)` | `#expect(a == b)` |
| `XCTAssertTrue(x)` | `#expect(x)` |
| `XCTAssertNil(x)` | `#expect(x == nil)` |
| `XCTAssertThrowsError` | `#expect(throws:)` |
| `setUp()` / `tearDown()` | `init()` / `deinit` |

### Enhancements
- `@Suite` for grouping related tests
- `@Test(arguments:)` for parameterized tests (e.g., contact initials, domain extraction)
- Native `async` test support — remove `XCTestExpectation` / `waitForExpectations`
- Typed error assertions where applicable

### Semantically Obsolete Tests (must rewrite, not just convert)
`AppCoordinatorTests` contains `testMailboxViewModelChangesForwardToCoordinator` and `testMailStoreChangesForwardToCoordinator` — these explicitly subscribe to `coordinator.objectWillChange` via Combine sinks to verify nested VM forwarding. After `@Observable` migration, `objectWillChange` no longer exists and the forwarding mechanism is removed entirely (`@Observable` handles granular tracking automatically). These tests must be **rewritten** to verify equivalent behavior under `@Observable` (e.g., that mutating nested VM state correctly updates dependent views), not just syntactically converted.

### Dead Code Removal
- All `XCTestCase` subclassing boilerplate
- `setUp()` / `tearDown()` methods
- `XCTestExpectation` patterns
- `test` prefix convention

## 6. Language Feature Modernization & Dead Code Cleanup

### Typed Throws (SE-0413)
- `GmailAPIError` is a well-defined enum → convert service methods from `throws` to `throws(GmailAPIError)`
- Callers get precise `catch` blocks without casting

### count(where:) Adoption
- Replace `.filter { predicate }.count` → `.count(where: { predicate })` throughout
- More efficient — no intermediate array allocation
- Note: available since Swift 5.9 (SE-0220), not new to 6.2 — but included here as a cleanup pass

### nonisolated on Types
- Pure data services or utility types that don't need MainActor → mark `nonisolated`

### Access-Level Imports
- `private import AppAuth` in OAuthService
- `private import Sparkle` in UpdaterViewModel
- Implementation-detail dependencies hidden from module interface

### Global Dead Code Audit
- Remove unused functions, types, extensions across all 135 files
- Remove commented-out code blocks
- Remove unused `import` statements
- Remove `#if DEBUG` code that's no longer useful
- Remove all remaining Combine imports/usage from observation
- Remove deprecated API usage flagged by macOS 26 SDK
- Audit singleton services — remove any that are unused or redundant

## Execution Strategy

Big Bang on a feature branch (`swift-6.2-migration`). Changes grouped into 6 parallel work streams matching the sections above. Each stream is independent enough for parallel subagent execution, with a final integration pass to resolve any cross-cutting issues.

### Build Verification
- Compile after each section's changes
- Run full test suite after Section 5 (testing migration)
- Final clean build + test pass before merge

### Risk Mitigation
- Baseline build verified clean (BUILD SUCCEEDED, 0 errors)
- Git branch provides rollback path
- Each section can be reverted independently if needed
