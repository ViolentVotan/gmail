# Services

Business logic, networking, and side effects. This is the **only** layer that talks to external APIs.

## Guidelines

- Services are **singletons** (`static let shared`) — stateless request handlers, not state containers.
- All API calls go through `GmailAPIClient` which handles auth tokens, rate limiting, and logging.
- Services return raw API models (`GmailMessage`, `GmailLabel`, etc.). They do NOT return UI models. Model types conform to `Sendable`.
- Error handling: throw typed errors (`throws(GmailAPIError)`) up to the ViewModel. Services don't show UI or set observable state.
- Services must be **account-aware**: every method takes `accountID` as parameter.
- **Concurrency**: I/O-bound service methods use `@concurrent` for off-MainActor execution. Default isolation is `MainActor` (approachable concurrency). `UndoActionManager` and `NetworkMonitor` use `@Observable` (not `ObservableObject`).
- No SwiftUI imports in pure service files.

## Subfolders

### `Auth/`
OAuth flow, token storage (Keychain), token refresh, token revocation. `OAuthService` handles the Google OAuth PKCE flow, `revokeToken(token:)` for sign-out revocation via Google's endpoint, and `refreshToken` with `invalid_grant` detection (`OAuthError.tokenRevoked`). Both `refreshToken` and `postForm` use typed throws (`throws(OAuthError)`); `OAuthError` includes `.networkError(any Error)` for wrapping `URLError` and similar transport failures. `postForm` retries on 5xx and network errors with exponential backoff (up to 3 attempts, `pow(2, attempt)` second delays; 4xx errors are not retried). `TokenStore` persists tokens securely (`save()` is `@concurrent async` — JSON encoding + AES-GCM encryption + Keychain write off MainActor; Keychain helpers are `nonisolated`). `AuthToken` is the token value type (`Codable`, `Sendable`) with `accessToken`, `refreshToken`, `expiresAt`, `tokenType`, `scope` properties and expiration tracking (60-second pre-refresh buffer via `isExpired`).

### `Protocols/`
- `MessageFetching` — Protocol abstracting the Gmail message API surface (`@MainActor`, `Sendable`, typed throws). Methods include `getProfile(accountID:)` for history ID retrieval and `batchModifyLabels(ids:add:remove:accountID:)` for bulk label operations. `GmailMessageService` conforms. Enables testing via mocks (`MockMessageFetching`).

### `Gmail/`
Gmail REST API wrappers. One service per domain:
- `GmailAPIClient` — HTTP layer (auth headers, base URL, per-account token refresh coalescing via unified `refreshAndRetry`, retry with exponential backoff + Retry-After on 429/500/503 capped at 64s, 401 auto-retry on standard and batch requests, generic `batchFetch<T>()` returning `BatchFetchResult<T>` (items + failedIDs) for batch GET operations (max 2 concurrent batches to avoid Gmail's per-user concurrent request limit) with `@concurrent fetchSingleBatch`; `batchFetch` and `retryRateLimitedParts` are themselves `@concurrent`; `requestURL()` for non-Gmail Google APIs; `Logger`-based diagnostics). `request()`/`rawRequest()`/`requestWithETag()`/`requestURL()`/`batchRequest()` are `@concurrent` to avoid unnecessary MainActor hops; `cachedValidToken` reads from a `Mutex<[String: AuthToken]>`-protected cache to skip MainActor when the token is valid, only hopping to MainActor for refresh. `sharedSession` (internal visibility) shared with `GmailProfileService`. `requestWithToken(url:token:)` is a `@concurrent nonisolated static` method for pre-sign-in flows (raw access token, full retry). `performWithETag` has full retry/backoff matching `perform()` (retries 429/500/502/503/504, respects Retry-After, preserves 304 passthrough). Per-part 429 retry in batch responses via `retryRateLimitedParts` (exponential backoff, max 3 attempts, `var activeToken` updated after 401 refresh for subsequent iterations). `batchFetch` returns failed IDs to callers for retry instead of silently dropping them. `uploadResumable(mimeData:threadID:accountID:)` sends large messages via Gmail's resumable upload protocol (initiate + PUT); both steps include `Accept-Encoding: gzip`. `perform()` and `performBatch()` track quota spend via `QuotaTracker.shared` (fire-and-forget — 5 units per request, 50 per batch). `GmailAPIError` includes `tokenRevoked` (mapped from `OAuthError.tokenRevoked` via `wrap()`; `refreshAndRetry` clears the revoked token from Keychain and in-memory cache before rethrowing), `dailyLimitExceeded`, and `domainPolicy` cases. Batch response parsing strips the `response-` prefix from Content-IDs (case-insensitive header matching per RFC 7230). `GmailPathBuilder` utility enum provides `queryAllowed` charset (internal), `pathAllowed` charset, `labelQueryParam`, `sendAsPath`.
- `GmailMessageService` — Messages, threads, mutations (trash, archive, star, labels), History API. Label modifications delegate to a single `modifyLabels` method (returns `fields="id,threadId,labelIds"`); `trashMessage` and `untrashMessage` also use `fields="id,threadId,labelIds"`. `batchModifyLabels(ids:add:remove:accountID:)` applies label changes in batches of 1000 (50 quota units per call). `batchDelete(ids:accountID:)` permanently deletes messages in batches of 1000 (50 quota units per batch vs 1 per message for sequential deletion); used by `OfflineActionQueue` for `.deletePermanently`. Uses `GmailPathBuilder.queryAllowed` for query parameter encoding. `getMessages` returns `(messages: [GmailMessage], failedIDs: [String])` tuple; format-specific `fields` via `MessageFields` enum (minimal: id/threadId/labelIds/internalDate; metadata: headers/snippet/labels; full: payload/snippet/labels).
- `GmailLabelService` — Label CRUD. `createLabel` and `updateLabel` include a `fields` parameter to select specific label properties in the response.
- `GmailProfileService` — Gmail profile info, user identity (OAuth userinfo), send-as aliases, signature management. `getUserInfo` delegates to `GmailAPIClient.requestWithToken` for pre-sign-in flows (shared retry logic, no duplicate URLSession). `updateSignature` includes a `fields` parameter scoping the response to `sendAsEmail,displayName,signature`.
- `GmailSendService` — Compose, send, draft CRUD (RFC 2822 MIME encoding with RFC 2047 header encoding, base64 Content-Transfer-Encoding for text parts). Outgoing messages include `Date:` and `Message-ID: <UUID@vik.app>` headers. Reply headers use separate `In-Reply-To` (single parent Message-ID) and `References` (full chain per RFC 2822 §3.6.4). Header construction is extracted into a `nonisolated private static func buildHeaders(...)` reused by both plain and multipart builders. Base64url encoding uses `Data.base64URLEncodedString()` extension. `buildReferencesChain(parentReferences:parentMessageID:)` assembles the References header from parent chain + Message-ID. `sendRaw(base64url:threadID:accountID:)` sends a pre-built RFC 2822 message (used by offline send queue).
- `GmailDraftService` — Draft fetch (single + batch via `batchFetch` with format-specific `fields` via `DraftFields` enum); `getDrafts` returns `(drafts: [GmailDraft], failedIDs: [String])` tuple. `createDraft`/`updateDraft` accept `bcc`, `inReplyTo`, `references`, and `threadID` parameters (threading headers preserve reply chain; threadID preserves thread association for reply drafts); both delegate to a shared `saveDraft` implementation (POST for create, PUT for update) and pass a `fields` parameter scoping the response payload. `sendDraft` passes `fields: "id"` to minimize response size. Used for quick reply draft loading.
- `GmailFilterService` — Gmail filter CRUD (list, create, delete filters). `createFilter` includes a `fields` parameter to scope the response to `id,criteria,action`.
- `GmailSendAs+BestAlias` — `Array<GmailSendAs>` extension with `bestAlias(toRecipients:ccRecipients:)` for selecting the most appropriate send-as alias when composing replies.
- `GmailModels` — All API response/request types (`Codable` structs). `GmailMessage` has a stored `headerMap: [String: String]` property built once during `init(from decoder:)` for O(1) header lookups (avoids rebuilding per access). `isSuspiciousSender` uses dot-boundary domain comparison. Static regexes (`receivedFromRegex`, `dkimDomainRegex`) use closure-initialized `let` with `preconditionFailure` guard.

### `Calendar/`
Google Calendar REST API v3 wrappers and sync engine. One service per domain:
- `CalendarAPIClient` — HTTP layer for Calendar API v3. `@MainActor` singleton. Delegates token management to `GmailAPIClient` (shared token bridge — no duplicate token refresh logic). `@concurrent` methods with `typed throws(CalendarAPIError)`. Base URL: `https://www.googleapis.com/calendar/v3`. Retries 401 with token refresh; exponential backoff on 429/403. Posts `insufficientPermissions` notification to trigger scope reauthorization UI.
- `CalendarAPIError` — Typed error enum: `invalidURL`, `unauthorized`, `offline`, `tokenRevoked`, `httpError`, `decodingError`, `encodingError`, `networkError`, `notFound`, `conflict(etag)`, `rateLimited(retryAfter)`, `gone`, `insufficientPermissions`. Includes `wrap(_:)` for bridging from `GmailAPIError`.
- `CalendarModels` — 18 `Codable` types mapping 1:1 to Calendar API v3 JSON. Response types (`CalendarEventListResponse`, `CalendarAPIEvent`, `CalendarAPICalendarListEntry`, etc.), nested types (`CalendarAPIConferenceData`, `CalendarAPIReminders`, etc.), and input types (`CalendarAPIEventInput`, `CalendarAPIAttendeeInput`).
- `CalendarEventService` — Events CRUD: `listEvents` (field mask + pagination), `syncEvents` (syncToken incremental), `getEvent`, `insertEvent`, `updateEvent` (Replaces an event (PUT) with If-Match etag + `conferenceDataVersion` support), `patchEvent` (private, used for RSVP), `deleteEvent`, `quickAdd` (natural language parsing), `respondToEvent` (convenience wrapper around `patchEvent`). All mutating methods accept `sendUpdates: String? = "all"` to control attendee notifications (Google Calendar API v3 `sendUpdates` parameter). Both `insertEvent` and `updateEvent` accept `conferenceDataVersion: Int?` for Google Meet link preservation.
- `CalendarListService` — CalendarList CRUD: `listCalendars` (paginated, returns entries + syncToken), `syncCalendars` (syncToken incremental). All methods use `fields` partial-response masks.
- `CalendarFreeBusyService` — `queryFreeBusy`: POST `/freeBusy` with RFC 3339 start/end dates.
- `CalendarSyncEngine` — `actor`, owned by `AppCoordinator` (peer to `FullSyncEngine`). Initial sync: calendarList → events per calendar (–30d to +90d window). Incremental sync: syncToken per calendar with 410 Gone recovery (falls back to full re-fetch). Calendar list uses incremental syncToken sync with deleted-entry handling (entries with `deleted == true` are removed from the local DB via `syncer.deleteCalendars`). Guards against offline state (`NetworkMonitor.isReachable`) to prevent error spam when connectivity is lost. All-day event date parsing uses an explicit Gregorian calendar (not `Calendar.current`) for correctness with non-Gregorian locale settings. Adaptive polling: 30s (calendar tab active), 60s (mail mode), 300s (background), 15s (post-edit for 2 minutes via `temporarilyTightenPolling`). Polling adjusts on app active/resign and view mode switch.
- `CalendarBackgroundSyncer` — Actor for bulk Calendar DB writes: `upsertCalendars`, `deleteCalendars`, `upsertEvents` (with attendee cascade), `deleteEvents`, `updateSyncToken`. Includes API → Record conversion helpers.
- `CalendarOfflineActionQueue` — Separate offline queue for calendar mutations via `PerAccountFileStore<CalendarOfflineAction>`. Actions: `createEvent`, `updateEvent`, `deleteEvent`, `rsvpEvent` — each carries `sendUpdates: String?` for attendee notification control during offline replay. Per-account reentrancy guard (`processingAccounts` set) prevents concurrent `processQueue` calls from duplicating API mutations. FIFO processing; stops on first failure (no partial-apply of a conflicting sequence). 409 Conflict on updates triggers automatic conflict resolution: re-fetches the event for a fresh etag and retries once, with a user-visible toast on failure.
